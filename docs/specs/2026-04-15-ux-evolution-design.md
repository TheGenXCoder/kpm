# KPM UX Evolution — Design Spec

**Date:** 2026-04-15
**Status:** Proposed (pending Grok review)
**Scope:** shell-init, auto-discovery, layered templates, XDG compliance, directory-bound templates with profile variables, include directive

---

## Overview

Five connected changes that make KPM feel native to the shell and project ecosystem. Moves KPM from "powerful but plumbing-heavy" to "belongs in my shell."

1. **XDG compliance** — config in `$XDG_CONFIG_HOME/kpm/`, data in `$XDG_DATA_HOME/kpm/`
2. **`kpm shell-init`** — one-line shell integration, no ugly eval chains
3. **Auto-discovery** — `kpm run -- claude` finds `claude.template` automatically
4. **Directory-bound templates with includes** — `${kms:include:customers/companyx/claude}`
5. **Profile variables** — `${kms:include:customers/{{profile:company_name}}/claude}`

---

## 1. XDG Compliance

### Current

```
~/.kpm/
  config.yaml
  certs/
  templates/
  sessions/
```

### Proposed

```
$XDG_CONFIG_HOME/kpm/              # defaults to ~/.config/kpm/
  config.yaml                      # KPM configuration
  templates/                       # template tree (user level)
    shell-env.template
    ai/
      claude.template
    customers/
      companyx/
        claude.template
        production/
          db.template

$XDG_DATA_HOME/kpm/                # defaults to ~/.local/share/kpm/
  certs/                           # mTLS certificates
    ca.crt
    client.crt
    client.key
  sessions/                        # active decrypt sessions
    s1a2b3/
      key
      socket
      pid
```

### Why separate

- **Config** (config.yaml, templates) — version-controlled, portable, works with stow/chezmoi/yadm
- **Data** (certs, sessions) — machine-specific, not version-controlled, ephemeral

### Fallback

If `~/.kpm/` exists, use it (backward compatibility). New installs go to XDG paths. `kpm init` creates XDG by default. Migration command: `kpm migrate-config` (future, not v0.1).

### Resolution order for config

1. `--config` flag (explicit)
2. `$KPM_CONFIG` env var
3. `$XDG_CONFIG_HOME/kpm/config.yaml`
4. `~/.config/kpm/config.yaml`
5. `~/.kpm/config.yaml` (legacy fallback)

### Resolution order for data

1. `$KPM_DATA` env var
2. `$XDG_DATA_HOME/kpm/`
3. `~/.local/share/kpm/`
4. `~/.kpm/` (legacy fallback)

## 2. `kpm shell-init`

### Current (ugly)

```bash
# .bashrc / .zshrc
eval $(kpm env --from ~/.kpm/templates/shell-env.template --output shell 2>/dev/null)
```

### Proposed (clean)

```bash
# .bashrc / .zshrc — one line, done forever
eval "$(kpm shell-init)"
```

Or the hook-file variant (no eval at all):

```bash
# Generated by kpm init
[ -f ~/.config/kpm/shell-hook.sh ] && source ~/.config/kpm/shell-hook.sh
```

### What `kpm shell-init` outputs

Minimal shell code that:

1. Checks if `$XDG_CONFIG_HOME/kpm/templates/shell-env.template` exists
2. If yes, loads ciphertext env vars from it
3. Starts the background decrypt listener
4. Sets `KPM_SESSION` and `KPM_DECRYPT_SOCK`
5. Exports a `KPM_LOADED=1` marker (for other tools to detect)

### Shell detection

`kpm shell-init` detects `$SHELL` or accepts `--shell=bash|zsh|fish`:

```bash
kpm shell-init              # auto-detect from $SHELL
kpm shell-init --shell=zsh  # explicit
kpm shell-init --shell=fish # fish-compatible output
```

### Performance

The shell-init output must be lightweight. It should NOT:
- Start a server
- Make network calls
- Do heavy computation

It SHOULD:
- Source a pre-generated hook file, or
- Output 5-10 lines of shell code that lazy-loads on first use

### Missing setup detection

If KPM is not configured (no config.yaml, no certs), `kpm shell-init` outputs nothing and prints a warning to stderr:

```
kpm: not configured. Run: kpm quickstart
```

This prevents breaking the shell on fresh machines.

## 3. Auto-Discovery for `kpm run`

### Current

```bash
kpm run --from .kpm/templates/claude.template -- claude "fix the bug"
```

### Proposed

```bash
kpm run -- claude "fix the bug"
```

### Resolution order

When `kpm run -- <command> [args]` is invoked without `--from`:

1. **Project command template:** `./.kpm/templates/<command>.template`
2. **User command template:** `$XDG_CONFIG_HOME/kpm/templates/<command>.template`
3. **Project directory template:** `./.kpm/templates/ai/<command>.template` (subdirectory search)
4. **User directory template:** `$XDG_CONFIG_HOME/kpm/templates/ai/<command>.template`
5. **`$KPM_RUN_SECRETS` env var** — comma-separated list of env var names to decrypt
6. **`--all` flag** — explicit opt-in to decrypt everything
7. **Error:** `No template found for 'claude'. Create .kpm/templates/claude.template or use --from`

### Command name extraction

- `kpm run -- claude "fix"` → looks for `claude.template`
- `kpm run -- /usr/bin/claude "fix"` → extracts `claude`, looks for `claude.template`
- `kpm run -- python script.py` → looks for `python.template`

### Scoping behavior

**Command templates are standalone.** They declare only the secrets the command needs. They do NOT automatically inherit `shell-env.template`. This prevents over-provisioning.

If a command template wants base secrets, it uses the include directive (see section 4):

```bash
# claude.template
${kms:include:shell-env}
CLAUDE_MODEL=claude-opus-4-6
```

### `--all` flag

Explicit opt-in to decrypt everything in the current session:

```bash
kpm run --all -- legacy-tool
```

This is the escape hatch for tools that need everything. It's never the default.

## 4. Directory-Bound Templates with Include Directive

### Template tree structure

Templates are organized as a directory tree under the templates root:

```
$XDG_CONFIG_HOME/kpm/templates/
  shell-env.template              # base shell secrets
  ai/
    claude.template               # default claude config
    cursor.template
  infra/
    terraform.template
    ansible.template
  db/
    postgres.template
    redis.template
  customers/
    companyx/
      claude.template             # CompanyX claude (different API key)
      production/
        db.template               # CompanyX prod DB
      staging/
        db.template               # CompanyX staging DB
    companyy/
      claude.template
      production/
        db.template
```

### Include directive

Templates can include other templates:

```bash
# ai/companyx/claude.template
${kms:include:ai/claude}                    # inherit base claude config
ANTHROPIC_API_KEY=${kms:companyx/anthropic-key}  # override with client key
```

**Include resolution:**
- Path is relative to the templates root (`$XDG_CONFIG_HOME/kpm/templates/`)
- `.template` suffix is added automatically
- Included template's entries are merged first, then the current template's entries override
- Circular includes are detected and error

**Multiple includes:**

```bash
${kms:include:shell-env}
${kms:include:infra/terraform}
AWS_REGION=us-west-2
```

### Project-level templates

Project-level templates live in `./.kpm/templates/` (relative to cwd). They follow the same structure:

```
~/clients/companyx/project-alpha/
  .kpm/
    config.yaml                   # profile: company_name: companyx
    templates/
      claude.template             # project-specific overrides
      shell-env.template          # project-specific env additions
```

### Merge order for includes

When multiple levels define the same template:

1. Included template content (base)
2. Current template content (overrides same keys, adds new keys)

When project and user level both have the same template:

1. User-level template (base)
2. Project-level template (overrides/adds)

## 5. Profile Variables

### Project config

`.kpm/config.yaml` in the project root (NOT from `$XDG_CONFIG_HOME` — only from `pwd`):

```yaml
profile:
  company_name: companyx
  environment: production
  region: us-west-2
```

### Template variable syntax

```bash
${kms:include:customers/{{profile:company_name}}/claude}
# resolves to: customers/companyx/claude (→ customers/companyx/claude.template)

${kms:include:customers/{{profile:company_name}}/{{profile:environment}}/db}
# resolves to: customers/companyx/production/db
```

### Use cases

**Same template, different clients:**

```bash
# Template: ai/client-claude.template
${kms:include:customers/{{profile:company_name}}/claude}
CLAUDE_MODEL={{profile:model:-claude-opus-4-6}}
```

```yaml
# ~/clients/companyx/.kpm/config.yaml
profile:
  company_name: companyx
  model: claude-sonnet-4-6
```

```yaml
# ~/clients/companyy/.kpm/config.yaml
profile:
  company_name: companyy
  model: claude-opus-4-6
```

Same template reference, different secrets and config per client.

**Hardcoded includes (no profile needed):**

For simpler cases, skip profiles entirely:

```bash
${kms:include:customers/companyx/claude}
```

Direct path, no variables. Works when you don't need the indirection.

### Profile variable resolution

Profile configs are **merged up the directory tree** — child overrides parent for the same key. This eliminates repetition across nested project structures.

**Resolution order (bottom-up merge):**

1. Walk from current working directory up to filesystem root
2. At each level, if `.kpm/config.yaml` exists, load its `profile:` section
3. Merge: deeper (child) values override shallower (parent) values for the same key
4. Final fallback: `$XDG_CONFIG_HOME/kpm/config.yaml` (user global defaults)
5. Variable syntax: `{{profile:key}}` with optional default `{{profile:key:-default}}`
6. Unresolved variables → clear error with the list of directories checked

**Example: multi-level org structure**

```
~/clients/acme/.kpm/config.yaml
  profile:
    customer: acme

~/clients/acme/us-east/.kpm/config.yaml
  profile:
    region: us-east

~/clients/acme/us-east/project-x/.kpm/config.yaml
  profile:
    project: project-x
    environment: staging
```

When working in `~/clients/acme/us-east/project-x/`:

```
Resolved profile:
  customer: acme          ← from ~/clients/acme/.kpm/config.yaml
  region: us-east         ← from ~/clients/acme/us-east/.kpm/config.yaml
  project: project-x     ← from ~/clients/acme/us-east/project-x/.kpm/config.yaml
  environment: staging    ← from ~/clients/acme/us-east/project-x/.kpm/config.yaml
```

One template handles all permutations:

```bash
# customers/base-db.template (reusable across all projects)
DB_HOST=${kms:{{profile:customer}}/{{profile:region}}/{{profile:project}}/db#host}
DB_PORT=${kms:{{profile:customer}}/{{profile:region}}/{{profile:project}}/db#port}
DB_PASSWORD=${kms:{{profile:customer}}/{{profile:region}}/{{profile:project}}/db#password}
```

**Why walking is safe for profiles (but not templates):**

- **Templates** control secret access — walking up could accidentally grant access to secrets from a parent scope. Templates stay explicit: project level or user level only.
- **Profile config** is plaintext metadata (company names, regions, project IDs). Walking up and merging is safe and expected — same pattern as `.gitconfig`, `.editorconfig`, `.npmrc`.

**Rule:** Templates don't walk. Profiles do.

Profile variables can be used in **both project-level and user-level templates.** This allows reusable user templates (e.g. `customers/{{profile:customer}}/{{profile:environment}}/db.template`) to adapt automatically based on the project context.

Profile values are **not secrets** — they are configuration (company names, environment names, region identifiers, project IDs, division codes). They are plaintext in the config file. Secrets use `${kms:...}` references; profiles use `{{profile:...}}`.

### Edge case handling

- No `.kpm/config.yaml` found at any level → fall back to global config only
- Profile key missing after full merge → clear, actionable error (never silently use empty string)
- Keep values simple for v0.1 — string values only, no nested objects or arrays
- Walk stops at filesystem root (or configurable stop marker like `.git` in the future)
- Circular references impossible (walking is strictly upward)

## Combined UX Flow

### Initial setup (once)

```bash
# Install
curl -sL kpm.catalyst9.ai/install | bash

# Quickstart (or connect to existing AgentKMS)
kpm quickstart

# Add your secrets
kpm add anthropic/api-key
kpm add github/deploy-pat --tags ci
kpm add companyx/anthropic-key --description "CompanyX Anthropic"

# Create templates
# (quickstart creates shell-env.template; user creates others as needed)
```

### Daily use

```bash
# .bashrc — one line
eval "$(kpm shell-init)"

# Shell has ciphertext for base secrets
echo $ANTHROPIC_API_KEY  # ENC[kpm:...]

# Run tools — auto-discovery by command name
kpm run -- claude "fix the bug"       # finds claude.template
kpm run -- terraform apply            # finds terraform.template

# In a client project
cd ~/clients/companyx/project-alpha
kpm run -- claude "analyze the codebase"
# → finds .kpm/templates/claude.template (project level)
# → includes customers/companyx/claude (via profile variable)
# → uses CompanyX's API key, not your personal one
```

### Multi-client workflow

```bash
# CompanyX project
cd ~/clients/companyx/webapp
cat .kpm/config.yaml
# profile:
#   company_name: companyx

kpm run -- claude "fix the auth bug"
# → resolves {{profile:company_name}} → companyx
# → includes customers/companyx/claude.template
# → ANTHROPIC_API_KEY = CompanyX's key

# CompanyY project  
cd ~/clients/companyy/data-pipeline
cat .kpm/config.yaml
# profile:
#   company_name: companyy

kpm run -- claude "optimize this query"
# → resolves {{profile:company_name}} → companyy
# → includes customers/companyy/claude.template
# → ANTHROPIC_API_KEY = CompanyY's key
```

Same `claude.template`, different secrets. Zero context-switching friction.

## Implementation Phases

### Phase A: XDG + shell-init (build now)

1. Update config resolution to check XDG paths first, `~/.kpm` fallback
2. Update data paths (certs, sessions) to use `$XDG_DATA_HOME/kpm/`
3. Implement `kpm shell-init` command (shell detection, hook output)
4. Update `kpm init` / `kpm quickstart` to create XDG structure
5. Update README and docs

### Phase B: Auto-discovery (build now)

6. `kpm run` extracts command name from args
7. Template search in `./.kpm/templates/` then `$XDG_CONFIG_HOME/kpm/templates/`
8. Subdirectory search (flat first, then `ai/`, `infra/`, etc.)
9. `$KPM_RUN_SECRETS` fallback
10. `--all` flag for explicit decrypt-everything
11. Clear error messages with suggestions

### Phase C: Include directive (build next)

12. Parse `${kms:include:path}` in template files
13. Resolve includes against templates root
14. Merge semantics (included = base, current = override)
15. Circular include detection
16. Tests for nested includes

### Phase D: Profile variables (build next)

17. Walk up directory tree collecting `.kpm/config.yaml` profile sections
18. Merge bottom-up (child overrides parent for same key)
19. Final fallback to `$XDG_CONFIG_HOME/kpm/config.yaml`
20. Parse `{{profile:key}}` and `{{profile:key:-default}}` in include paths and template references
21. Error on unresolved variables with list of directories checked
22. Tests for multi-level org structure (customer → region → project → environment)

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| XDG default, ~/.kpm fallback | XDG | Standard, works with stow/chezmoi, community expects it |
| Config vs data split | XDG_CONFIG_HOME vs XDG_DATA_HOME | Templates are config (portable), certs/sessions are data (machine-specific) |
| shell-init over eval chain | shell-init | Matches rbenv/direnv/starship pattern, hides plumbing |
| Command templates standalone | Not inheriting shell-env | Prevents over-provisioning, explicit includes for composition |
| Include directive | ${kms:include:path} | Consistent with existing ${kms:...} syntax |
| Profile variables | {{profile:key}} | Distinct from ${kms:...} to avoid ambiguity |
| Templates: no directory walking | Explicit project or user level | Prevents accidental secret access from parent scope |
| Profile config: walk up and merge | Bottom-up merge, child overrides parent | Eliminates repetition in nested org structures. Safe because profiles are plaintext metadata, not secrets. Same pattern as .gitconfig, .editorconfig, .npmrc |
| Template path convention | Directory tree | Mirrors filesystem conventions, scales to multi-client |
